# 기여 가이드 (Contributing)

세종마켓 백엔드 프로젝트에 기여하기 전에 아래 협업 규칙을 확인해 주세요.

## 1) 브랜치 전략 (Git Flow)

브랜치 전략으로 **Git Flow**를 사용합니다.

| 브랜치 | 용도 |
|--------|------|
| `master` | 최종 제출용 |
| `develop` | 통합 개발용 |
| `feature/*` | 기능 단위 개발 브랜치 |

### 권장 작업 흐름

1. `develop`에서 `feature/기능명` 브랜치 생성
2. 기능 구현 및 테스트
3. PR(Pull Request) 생성 후 코드 리뷰
4. 리뷰 반영 후 `develop`에 병합

## 2) 커밋 규칙

커밋 메시지는 아래 Prefix를 사용합니다.

| Prefix | 설명 |
|--------|------|
| `Create` | 새 파일 생성 |
| `Delete` | 파일/코드 삭제 |
| `Modify` | 단순 수정 |
| `Feat` | 기능 구현 |
| `Test` | 테스트 추가 |
| `Bug` | 버그 제보 |
| `Fix` | 버그 수정 |

### 커밋 메시지 예시

```
Feat: 상품 등록 API 구현
Fix: 상품 상태 변경 시 권한 검증 오류 수정
Test: UserService 회원가입 테스트 추가
```

## 3) PR(Pull Request) 규칙

PR 작성 시 아래 항목을 포함해 주세요.

- **변경 목적**: 왜 이 변경이 필요한지
- **핵심 변경 사항**: 무엇을 수정/추가했는지
- **테스트 결과**: 로컬 테스트 또는 API 테스트 결과
- **리뷰 포인트**: 리뷰어가 중점적으로 확인하면 좋을 부분

## 4) 코딩 규칙

- 변수명·필드명은 **카멜 케이스(camelCase)** 를 사용합니다.
- 클래스명은 Java 관례에 따라 **파스칼 케이스(PascalCase)** 를 사용합니다.

### 계층 구조 준수

프로젝트는 Controller → Service → Repository 계층 구조를 따릅니다.  
각 계층의 역할과 디렉토리 구조는 [README.md](./README.md#4-프로젝트-구조)를 참고해 주세요.
